# MVP Options Format

When the PRD involves build scope decisions, present multiple well-articulated MVP options from narrowest to widest. This was called "EXCEPTIONALLY HELPFUL" — it's a signature output of this skill.

---

## Gold Standard Format

Present 3-4 options, narrowest to widest:

```markdown
**MVP 1: [Short name — punchy description]**
- [2-4 bullets on what's built]
- **What it unblocks:** [value delivered]
- **What it doesn't do:** [explicit limitations]
- **Effort:** [Small/Medium/Large — with a why]

**MVP 2: [Short name — punchy description]**
- [2-4 bullets on what's built]
- **What it unblocks:** [value delivered]
- **What it doesn't do:** [explicit limitations]
- **Effort:** [Small/Medium/Large — with a why]

**MVP 3: [Short name — punchy description]**
- [2-4 bullets on what's built]
- **What it unblocks:** [value delivered]
- **What it doesn't do:** [explicit limitations]
- **Effort:** [Small/Medium/Large — with a why]
```

End with your read on the tradeoffs — which option you'd recommend and why.

---

## Why This Matters

- Helps the decision-maker scope quickly
- Gives the team shared vocabulary ("we're building MVP 2" vs ambiguous scope)
- "What it doesn't do" prevents scope creep after the decision
- Effort sizing is relative (Small/Medium/Large), not in hours or sprints

## Rules

1. Options must be strictly ordered from narrowest to widest scope
2. Each option must be independently viable — not "MVP 1 + this extra thing"
3. "What it doesn't do" is mandatory — this is where scope boundaries live
4. Effort must include a "why" — "Large — because it requires a new data pipeline" not just "Large"
5. Present the options, then your recommendation. Let the user decide.
